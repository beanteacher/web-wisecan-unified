# 기능 정의서 — WiseCan 통합 메시징 서비스

> 작성일 2026-06-01

---

## 0. 읽는 법

- **단위는 "사용자 액션"** (예: "회원가입한다", "발송한다", "충전한다"). RQ-ID 단위 X.
- 각 액션 카드는 동일 7개 필드: **트리거 / 사전조건 / 정상 흐름 / 예외 / 권한 / 데이터 영향 / 공수**. 마지막 줄에 `RQ-ID 매핑` 트레이스.
- **공수 등급** — S(≤ 3 man-day) / M(3~10 man-day) / L(> 10 man-day). 추정치는 1차 가설이며 스프린트 계획서(`04_PROJECT_PLAN.md`)에서 보정한다.
- **권한 코드** — `MEMBER` / `COMPANY_MASTER` / `COMPANY_SUB_ACCOUNT` / `ADMIN` / `SUPER_ADMIN` / `익명`. (`COMPANY_SUB_ACCOUNT` 은 `MEMBER` 와 분리된 발송 전용 하위 계정 — 데이터 모델 `05 §2.1`) 키 스코프는 `send`, `history:read`, `callback:read`, `callback:manage`, `key:read`, `balance:read`, `template:read`, `brand:read` 등.

## 1. 회원 가입·인증

### 1.1. 개인 회원으로 가입한다
- **트리거** — 비회원이 `/signup` 진입 후 "회원가입" 클릭
- **사전조건** — 휴대폰 본인 인증 가능, 동일 명의 개인 회원 미존재
- **정상 흐름** — 휴대폰 본인 인증 → 이메일·비밀번호·이름 입력 → 약관 동의(필수/선택 분리) → 가입 완료 → 즉시 로그인. 이메일 인증 메일은 비동기 발송, 결제·세금계산서·SDK 키 메일은 인증 후 활성화
- **예외** — (a) 동일 명의 개인 회원 존재 → 차단. (b) 동일 명의가 회사 하위 계정으로 소속 잔존(본인인증 명의가 활성 회사 소속) → "○○ 회사에 소속되어 있어 가입할 수 없습니다. 해당 회사에서 퇴사(삭제) 처리가 완료되어야 신규 가입이 가능합니다" 안내·차단 (퇴사 처리 주체·발신번호 연쇄 삭제는 §3.3·§4.4). (c) 사업자 회원만 존재 → "스팸 발송 시 모든 계정 정지 가능" 고지 + 명시 확인 후 진행
- **부가(회사 직원의 개인 가입)** — 회사 하위 계정 사용자가 본인 명의 개인 회원이 되려 할 때도 별도 서류 심사 없이 본 일반 가입 절차(휴대폰 본인 인증 + 개인 서류 + 약관 동의)를 그대로 거쳐 정식 개인 `MEMBER` 로 생성한다. 발송 전용 하위 계정과 분리된 별개 계정이다
- **권한** — 익명 → `MEMBER`
- **데이터 영향** — `accounts` INSERT, 약관 동의 이력, 본인 인증 정보 봉인
- **공수** — M
- `RQ-AUTH-001~005, 007~009`

### 1.2. 사업자 회원으로 전환한다
- **트리거** — 가입한 개인 회원이 `/signup/business` 진입 (사업자 전환 신청)
- **사전조건** — 개인 회원 가입·로그인 완료, 대표자 휴대폰 본인 인증, 사업자 등록증·대표자 신분증 보유
- **정상 흐름** — 사업자 정보 입력(법인번호·상호·대표자명) → 사업자 등록증 + 대표자 신분증 업로드 → 신청 완료(상태 = `SUBMITTED`) → 운영자 심사 대기 → 승인 시 기존 계정에 회사 마스터 권한 부여 + `COMPANY` 연결 (신규 계정 생성 없음)
- **예외** — (a) 동일 명의 사업자 존재 → 신규 허용하되 이전 가입 기본 정보 삭제. (b) 위조 의심 신호(반복 신청·공공기관 명의·동일 사업자번호 다중 신청) → 자동 재검증 큐로 라우팅. (c) 반려 → 사유 조회 후 보완 재제출 가능
- **권한** — `MEMBER`(개인) → `COMPANY_MASTER`(승인 후)
- **데이터 영향** — `business_application`, `documents`(서류 OCR + DB), 통화 녹음(승인 단계), 기존 `MEMBER` UPDATE(role·company_id)
- **공수** — L
- `RQ-AUTH-101~106, 010, 011`

### 1.3. 로그인한다 / 2차 인증한다
- **트리거** — 회원이 `/login` 진입
- **정상 흐름** — 이메일 + 비밀번호 입력 → (2차 인증 활성 시) 문자 또는 OTP 코드 입력 → 세션 발급(회원 12h)
- **예외** — 5회 실패 → 15분 잠금 + CAPTCHA. 신뢰 IP 등록된 경우 2차 인증 자동 패스
- **권한** — `MEMBER` / `COMPANY_*`
- **공수** — S (2차 인증 포함 M)
- `RQ-AUTH-301~311`

### 1.4. 비밀번호 / 아이디를 찾는다
- **트리거** — `/find-id` 또는 `/reset-password`
- **정상 흐름** — 본인 인증 → 등록 이메일로 재설정 링크 발송 (비밀번호) / 아이디 노출 (아이디 찾기)
- **공수** — S
- `RQ-AUTH-303, 304`

## 2. 비회원 체험

### 2.1. 체험 모드에 진입한다
- **트리거** — 비회원이 `/` 의 "체험하기" 버튼 클릭
- **사전조건** — 동일 IP/디바이스 최근 N분 내 체험 세션 발급 횟수 < 임계치
- **정상 흐름** — 익명 체험 세션 발급 → 더미 회원사 컨텍스트(더미 발신번호·키·발송 이력·잔액·결제 이력·카카오 템플릿·RCS 브랜드) 사전 적재 → 회원과 동일 UI 진입 → 상시 워터마크 "체험 모드 — 실제 발송·결제는 일어나지 않습니다"
- **예외** — 어뷰징 차단 임계 초과 → 차단 안내
- **권한** — 익명 + 체험 세션 토큰
- **데이터 영향** — 체험 전용 격리 영역에만 적재(운영 DB 무영향)
- **공수** — L (더미 데이터 사전 적재 모듈이 큼)
- `RQ-TRY-001~004, 008, 009, 016`

### 2.2. 체험 모드에서 발송·결제를 시도한다
- **트리거** — 체험 세션 사용자가 발송 버튼 또는 충전 버튼 클릭
- **정상 흐름** — 시스템이 외부 송출·결제·세금계산서·카카오 등록을 차단하고 가상 결과만 반환
- **권한** — 체험 세션
- **공수** — M
- `RQ-TRY-005, 006, 007`

### 2.3. 체험 모드를 종료하거나 가입 전환한다
- **트리거** — 가입 CTA 클릭 또는 무활동 만료 또는 사용자 종료
- **정상 흐름** — 만료/종료 시 체험 데이터 폐기. 가입 완료해도 체험 더미 데이터는 회원 계정으로 이관하지 않음
- **공수** — S
- `RQ-TRY-010~014`

## 3. 회사 계정 / 하위 계정 (회사 마스터 — 권한 부여형)

### 3.1. 하위 계정을 생성·관리한다
- **트리거** — `COMPANY_MASTER` 권한 보유자가 `/company/members` 진입
- **정상 흐름** — 하위 계정 생성(아이디·이름·휴대폰·기능 권한 지정) → 비활성화 / 삭제(soft) / 권한 수정. 하위 계정 삭제(퇴사 처리) 시 그 계정 명의로 등록된 발신번호는 자동 연쇄 삭제(§4.4)되며, 삭제 완료 후 해당 명의의 개인 신규가입 차단이 해제된다(§1.1 예외 b)
- **하위 계정 독립 구조** — 하위 계정(`COMPANY_SUB_ACCOUNT`)은 `MEMBER` 와 분리된 발송 전용 계정으로, 본인인증·개인정보(약관) 동의 없이 로그인·발송·이력 조회만 가능. 발신번호·API 키를 독립 소유하지 않고 회사 자원을 공유(O2)하며, 회사 마스터 또는 본인이 언제나 삭제 가능
- **권한** — `COMPANY_MASTER` 만
- **데이터 영향** — `COMPANY_SUB_ACCOUNT`(company_id) INSERT/UPDATE, `COMPANY_INVITATION`(초대), 기능 권한 매핑 (`MEMBER` 와 분리된 독립 테이블 — 05 §2.1)
- **공수** — M
- `RQ-MEMBER-101~109`

### 3.2. 회사 마스터 권한을 부여·회수·이관한다
- **트리거** — 회사 마스터가 `/company/master-roles` 진입
- **정상 흐름** — 일반 계정 1건 선택 → 권한 부여 / 회수 / 이관(퇴사 대응). 이력 자동 기록
- **권한** — `COMPANY_MASTER` 만
- **공수** — S
- `RQ-MEMBER-201~204`

### 3.3. 본인 정보를 수정·탈퇴한다
- **정상 흐름** — `/profile`에서 휴대폰·이름·비밀번호 수정 (이메일 수정 불가). `/withdrawal`에서 탈퇴 신청 → 운영자 처리 후 잔액 정산·발신번호 회수
- **하위 계정 본인 퇴사** — 회사 하위 계정 사용자는 회사 마스터를 거치지 않고 본인이 직접 퇴사(삭제) 처리할 수 있다. 처리 시 본인 명의 발신번호가 연쇄 삭제(§4.4)되며, 이후 동일 명의로 개인 신규가입이 가능해진다(§1.1 예외 b)
- **공수** — S
- `RQ-MEMBER-001~004`

## 4. 발신번호

### 4.1. 개인 휴대폰 발신번호를 즉시 등록한다
- **트리거** — `MEMBER`가 `/callbacks/registration` 진입
- **사전조건** — 등록 번호 = 본인 인증 휴대폰 번호와 일치
- **정상 흐름** — 번호·용도 설명 입력 → 본인 인증 정보로 즉시 검증 통과 → 상태 `REGISTERED`
- **예외** — 본인 휴대폰 외 번호 입력 시 §4.2로 분기
- **권한** — `MEMBER` + `callback:manage` 스코프
- **공수** — M
- `RQ-CALLBACK-001, 004, 005`

### 4.2. 개인 비-휴대폰 번호를 등록한다
- **사전조건** — 인터넷전화·일반 유선 등 비-휴대폰 번호. 별도 소유 확인 본인인증 수단 보유
- **정상 흐름** — 번호·용도 설명 입력 → 통신사 명의 확인 등 별도 본인인증 → 상태 `REGISTERED`
- **공수** — M
- `RQ-CALLBACK-009`

### 4.3. 임직원 / 법인 대표번호를 등록한다 (서류 심사형)
- **트리거** — `COMPANY_MASTER` 가 임직원 명의 번호 또는 법인 대표번호를 회사 자원으로 등록 (하위 계정은 발신번호를 독립 소유하지 않고 회사 자원을 공유 — O2)
- **사전조건** — 임직원 = 재직증명서 + 본인 통신서비스 이용 증명서. 법인 = 법인 명의 증빙
- **정상 흐름** — 번호 + 용도 설명 + 서류 업로드(PDF/JPG ≤ 10MB) → 시스템 OCR + 위조 의심 자동 검증 → 운영자 심사 큐 → 승인 시 상태 `REGISTERED`(재직증명서 무조건 동봉). 상태 `SUBMITTED → UNDER_REVIEW → REGISTERED`
- **예외** — 위조 의심 → 자동 재검증 큐. 반려 → 사유 통보 후 보완 재제출
- **권한** — `COMPANY_MASTER` + `callback:manage`
- **공수** — L
- `RQ-CALLBACK-002, 003, 008, 010`

### 4.4. 발신번호를 삭제한다
- **트리거** — 회원이 발신번호 목록에서 삭제 (또는 회사 마스터가 직원 발신번호 일괄 처리)
- **정상 흐름** — 삭제 확인 → 상태 `DELETED` (`CALLBACK_LOG` 사건 기록)
- **연쇄 삭제** — 직원(하위 계정)이 회사 마스터 또는 본인에 의해 퇴사(삭제)되면, 그 직원이 등록한 발신번호는 별도 조작 없이 자동으로 `DELETED` 연쇄 처리된다 (회원 해지 연쇄 §12.3 과 동일 메커니즘)
- **공수** — S
- `RQ-CALLBACK-103`

## 5. API Key

### 5.1. 테스트 키를 발급 신청한다
- **트리거** — 회원이 `/keys` 에서 "Test Key 발급" 신청
- **정상 흐름** — 신청 → 운영자 검토 큐(`/admin/review/keys`) → 운영자 승인 시점에 키 발급 + 채널별 테스트 발송 한도 자동 부여(예: 채널당 5건). 결과는 메일·인앱으로 통보
- **예외** — 반려 시 사유 통보
- **권한** — `MEMBER` / `COMPANY_*`. 발급은 `ADMIN` (RQ-ADMIN-603)
- **공수** — M
- `RQ-KEY-001, 005, 012, RQ-TEST-005·006`

### 5.2. 운영 키로 전환한다
- **트리거** — 회원이 `/keys` 의 테스트 키에서 "운영 키로 전환" 신청
- **정상 흐름** — 신청 → 운영자 검토 큐. 검토 항목: 가입 심사 완료, 발신번호 등록(`REGISTERED`), 결제 수단·잔액, 채널 등록 충족, 어뷰징 신호. 승인 시 운영 키 즉시 활성화 + 실사용 캐시 차감으로 전환
- **권한** — 신청은 회원, 승인은 `ADMIN`
- **공수** — M
- `RQ-KEY-008, 012, RQ-ADMIN-603`

### 5.3. 키 스코프·발신번호 제한·일일 한도를 설정한다
- **트리거** — 키 발급 또는 키 수정
- **정상 흐름** — 스코프 카탈로그(12종) 다중 선택 → 발신번호 화이트리스트 → 일일 발송 한도 → 조회 범위 `scope:key`(기본) 또는 `scope:member`(옵션) 토글
- **예외** — 미허용 스코프 호출 → `SCOPE_NOT_GRANTED` 403
- **권한** — 키 소유 회원 (회사 키는 `COMPANY_MASTER`)
- **공수** — M
- `RQ-KEY-101~105, 201~205`

### 5.4. 키를 폐기·재발급한다
- **정상 흐름** — 폐기 → 즉시 무효(1초 내 모든 노드 캐시 무효화). 재발급 → 새 키 발급 시 기존 키 즉시 무효
- **권한** — 키 소유 회원. 운영자가 노출 의심 키 강제 폐기 가능(`RQ-ADMIN-207`)
- **공수** — S
- `RQ-KEY-006, 007`

## 6. 발송 — 웹 콘솔

### 6.1. 단건 메시지를 보낸다 (SMS / LMS / MMS / 카카오 알림톡 / RCS)
- **트리거** — 회원이 `/messages/{channel}` 진입
- **사전조건** — 등록(`REGISTERED`) 발신번호 보유, 잔액 또는 자동충전/후불, (카카오) 승인된 템플릿, (RCS) 등록된 브랜드
- **정상 흐름** — 발신번호 선택 → 수신번호 입력 → 본문 작성 → (광고 의무 표기 검사) → (스팸 키워드 검사) → 잔액 사전 평가 → 발송 테이블 적재 → 발송 ID + ACCEPTED 응답
- **예외** — 발신번호 미등록(`CALLER_NOT_REGISTERED`), 잔액 부족(§11로 분기), 스팸 키워드(`SPAM_KEYWORD_DETECTED`), 광고 표기 누락(`AD_DISCLOSURE_MISSING`), 채널 길이 초과(`TEXT_TOO_LONG`)
- **권한** — `send` 또는 `send:{channel}` 스코프
- **데이터 영향** — `send_table` INSERT, `cash_balance` 차감 (FIFO + 만료 임박 우선)
- **공수** — L (5채널 × 검증 파이프라인)
- `RQ-SEND-001~005, 008, 009, RQ-PAY-006·011·013`

### 6.2. 일괄 발송한다 (CSV)
- **트리거** — `/messages/batch` 에서 CSV 업로드

## 11. 잔액 부족 분기 처리 (횡단)

### 11.1. 발송 적재 전 사전 평가 → 분기
