2. 변경 항목 체크리스트

C1. 사업자 회원 가입 → 전환

현재 — /signup/business 에서 처음부터 사업자로 가입. BUSINESS_APPLICATION 은 가입 신청 큐. MEMBER INSERT 와 동시에 신청 INSERT.

변경 방향

  • 개인 회원으로 먼저 가입 후 사업자 전환 신청 → 승인 시 COMPANY 생성 + 기존 MEMBER.role = COMPANY_MASTER + company_id 부여.
  • BUSINESS_APPLICATION 의 의미를 "가입 신청" → "전환 신청" 으로 재서술(신규 MEMBER 생성 트랜잭션 제거).

C2. 회사 직원의 개인 회원 전환 (일반 회원가입과 동일 절차)

현재 — 하위 계정 → 개인 회원 전환 경로 정의 없음.

변경 방향

  • 회사 직원이 개인 회원이 되려 할 때 별도 예외 서류 심사를 두지 않는다.
  • 대신 일반 회원가입과 동일한 절차(휴대폰 본인인증 + 개인 서류 + 약관 동의 등)를 그대로 거쳐 정식 개인 MEMBER 로 생성한다.
  • 즉 하위 계정(COMPANY_SUB_ACCOUNT, 인증·약관 동의 없는 계정)은 개인 회원이 되려면 §1.1 일반 가입 흐름을 정상 완료해야 한다.

C3. 퇴사자 신규가입 처리 (서류 + 회사 유선 → 회사 측 삭제 → 발신번호 삭제)

현재 — 퇴사 후 개인 신규가입 시 기존 회사 소속 잔존 처리·발신번호 연쇄 삭제 흐름 미정의.

변경 방향 (O1 결정 반영)

  • 퇴사자가 개인으로 신규가입을 시도하면 "○○ 회사에 소속되어 있어 가입 불가" 안내·차단 — 해당 회사에서 퇴사(삭제) 처리가 완료되어야 신규 가입 가능.
  • 퇴사(삭제) 처리는 회사(마스터)와 본인(개인) 둘 다 직접 가능하며, 처리되면 그 직원의 발신번호도 연쇄 삭제.

C4. 발신번호 이력 — CALLBACK_OWNERSHIP_LOG 제거, CALLBACK_LOG 로 대체 테이블 삭제

현재 — CALLBACK_OWNERSHIP_LOG 가 번호별 소유권 이관의 단일 진실 원천(from/to_callback_id, reason 5종). 소유권 이관 모델 전제.

변경 방향

  • CALLBACK_OWNERSHIP_LOG 테이블 제거 — 소유권이 변경될 일은 없으므로 이관 개념 폐기.
  • 발신번호 등록 관련 이력은 CALLBACK_LOG 히스토리 테이블로 대체(등록·심사·삭제 등 사건 기록).

C5. 발신번호 설명 필드 추가

현재 — CALLBACK 에 용도 설명 컬럼 없음.

변경 방향 — "무슨 발신번호인지" 설명(용도/라벨) 컬럼을 CALLBACK 에 추가.

C6. KISA 테이블·컬럼 제거 테이블 삭제

현재 — KISA_REGISTRATION 테이블 + CALLBACK.kisa_status/kisa_ref_id 컬럼. 도입부·Redis 키(lock:kisa, kisa:cache, caller:registered)·검증 게이트·INV·인덱스·§14 캐시 전략 전반에 KISA 언급.

변경 방향 — KISA 관련 테이블·컬럼·행을 모두 제거. (발신번호 "등록됨" 상태 개념 자체는 유지하되 외부 KISA 연계 표현만 제거.)

C7. member_type 제거 — role + company_id 로만 식별

현재 — MEMBER.member_type = enum(PERSONAL, BUSINESS). 페르소나는 role × member_type 조합으로 식별.

변경 방향

  • member_type 컬럼 제거. 식별은 role + company_id NULL 여부만 사용.
  • 개인 = company_id IS NULL, 회사 마스터 = role=COMPANY_MASTER & company_id NOT NULL.

C8. TERM_AGREEMENT 이력 테이블 + 필수동의 enum 테이블 신설

현재 — TERM_AGREEMENT 단일 테이블(required/agreed CHAR(1) Y/N). 동의 변경 이력 없음.

변경 방향

  • TERM_AGREEMENT_LOG 이력 테이블 신설(가입·재동의·철회·버전 갱신 시 1행).
  • 필수 동의여부를 enum 으로 표현(필수 여부 자체는 C9 의 TERM_CODE 로 이동).

C9. TERM_CODE 테이블 + 이력 + 파일 보관 테이블 신설

현재 — 약관 메타(코드·버전·필수 여부)가 TERM_AGREEMENT 행마다 중복 보관. 약관 마스터·원문 보관 없음.

변경 방향

  • TERM_CODE 신설 — 시행 날짜 / 고시 날짜 / 약관 정보 / 필수 여부 보유. 필수 여부의 단일 소유.
  • TERM_CODE_LOG 개정 이력 + 약관 원문은 파일로 보관(경로만 DB).
  • TERM_AGREEMENT 는 TERM_CODE 를 참조하고 필수 여부를 직접 보유하지 않음.

C10. 사업자 하위 회원 독립 구조 — MEMBER 에서 분리 테이블 신설

현재 — 하위 계정이 MEMBER 테이블에 role=COMPANY_MEMBER 로 들어감. 초대(COMPANY_INVITATION) 수락 시 MEMBER INSERT. 마스터 자동 전이 후보로 하위 멤버 사용.

변경 방향

  • 하위 회원을 MEMBER 에서 분리해 독립 테이블(COMPANY_SUB_ACCOUNT 등)로 신설 — 본인인증·약관 동의 없이 로그인 · 발송 · 이력 조회만 가능.
  • 하위 계정은 언제나 삭제 가능(soft/hard) — MEMBER 와 다른 라이프사이클.
  • (O2 결정) 하위 계정은 발신번호·키를 독립 소유하지 않고 회사 자원을 공유. 개인 명의로 독립 소유하려면 개인 회원 전환(C2) 필요.
  • (O3 결정) 마스터 모델은 현행 유지 — §2.3 마스터 1명 강제·라이프사이클 정책 변경 없음.
  • MEMBER.role enum 에서 COMPANY_MEMBER 제거.
