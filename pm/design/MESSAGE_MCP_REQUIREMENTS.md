# MCP 문자 서비스 요구사항 명세서

> 작성일: 2026-04-29
> 상태: 1차 초안

## 사용자 역할 정의

본 명세서에서 사용하는 사용자 역할은 다음과 같이 명확히 분리한다.

| 역할 | 정의 | 권한 범위 |
|---|---|---|
| **비회원** | 가입 전 사용자 | 회원가입, FAQ 조회 |
| **회원(개인)** | 본인 인증으로 가입한 개인 사용자 | 자신의 발신번호·키·발송·결제 |
| **회원(사업자) — 회사 마스터** | 사업자 회원사의 오너 계정 (회사 단위 1인 또는 다수, A/B안 분기) | 회사 단위 발신번호·결제·하위 계정 관리. 시스템 차원 권한은 없음 |
| **회원(사업자) — 하위 계정** | 회사 마스터가 생성한 직원 계정 | 회사 마스터가 부여한 권한 범위 내 발송·키 발급 등 |
| **외부 개발자** | SDK/MCP를 사용하는 시스템 통합 사용자 | API Key 인증으로 발송 |
| **시스템 운영자(슈퍼 어드민)** | 본 서비스를 운영하는 회사의 총괄 관리자 | 모든 운영 권한 보유. 다른 운영자 계정 발급·권한 설정 |
| **시스템 운영자(일반)** | 슈퍼 어드민이 권한을 위임한 운영 담당자 | 위임된 권한만 (심사·CS·환불 등 분담) |

> ⚠ "마스터"라는 단어를 중의적으로 사용하지 않는다. 회사 단위 오너는 **"회사 마스터"**, 서비스 총괄은 **"시스템 운영자"** 또는 **"슈퍼 어드민"** 으로 표기한다.

## ID 체계

| Prefix | 분류 | 설명 |
|--------|-------|------|
| `RQ-AUTH` | 회원가입/로그인 | 회원가입, 본인인증, 서류 제출, 로그인, 2차인증 |
| `RQ-MEMBER` | 회사 마스터/하위 계정 | 고객사 내부의 회사 마스터, 하위 계정 관리 |
| `RQ-ADMIN` | 시스템 운영자 | 서비스 총괄 — 회원 심사/승인, 발송 통제, 환불 처리, 운영 설정 |
| `RQ-CALLER` | 발신번호 | 발신번호 등록·관리·검증 |
| `RQ-KEY` | API Key | 키 발급·관리·전환·권한 |
| `RQ-SDK` | SDK / MCP | SDK 다운로드, MCP 가이드, 매뉴얼 |
| `RQ-SEND` | 발송 | 채널별 메시지 발송, 발송 이력 |
| `RQ-TPL` | 카카오/RCS 템플릿 | 브랜드·템플릿 등록·이관 |
| `RQ-PAY` | 결제/캐시 | 충전, 자동충전, 후불, 구독, 환불, 세금 |
| `RQ-TEST` | 테스트 환경 | 테스트 포인트 |
| `RQ-SEC` | 보안/스팸 | 스팸 필터, 이상 패턴, 한도 차단 |
| `RQ-OPS` | 고객 응대 | 챗봇, FAQ, 1:1 문의 |
| `RQ-ARCH` | 시스템/아키텍처 | 채널 통합, 발송 흐름, 인코딩 |

엑셀 변환 시 ID 끝 3자리는 일련번호로 사용한다.

## 3. 시스템 운영자 (RQ-ADMIN)

> 시스템 운영자는 본 서비스를 운영하는 회사의 직원이며, 회원사를 관리·통제·지원한다.
> 슈퍼 어드민과 일반 운영자(권한 위임)로 분리된다.

### 3.1. 운영자 계정 / 권한 관리

| RQ-ID | 분류 | 요구사항명 | 요구사항 내용 | 비고 |
|---|---|---|---|---|
| RQ-ADMIN-001 | 운영자계정 | 운영자 로그인 | 운영자는 별도 어드민 페이지에서 로그인할 수 있다. | 회원 도메인과 분리 |

### §A. 회사 마스터 권한 구조 (RQ-MEMBER-201~303)

> 본 분기는 **회사(고객사) 내부의 마스터 계정 모델**에 대한 의사결정이며, 시스템 운영자(`RQ-ADMIN-*`)와는 무관하다.

| 항목 | A안 — 권한 부여형 | B안 — 트리 종속형 |
|---|---|---|
| **개념** | 회사 내에 별도의 마스터 계정을 두지 않고, 일반 계정 중 특정 사용자에게 회사 마스터 권한을 부여한다. | 회사 마스터 계정 아래 하위 계정이 트리 구조로 종속된다. |
| **DB 스키마** | `accounts` 단일 테이블 + `account_role`(주체, 역할) 다대다 매핑. 회사 식별은 `company_id` 컬럼. | `companies` 1차 테이블 + `accounts.company_id` FK + `accounts.is_company_master` 플래그. 트리 깊이 1단 고정. |
| **발신번호·키 종속** | 발신번호·API Key는 `company_id` 단위 + 등록자(`account_id`) 기록. 권한 보유자만 사용. | 발신번호·API Key는 회사 마스터 계정에 직접 종속. 하위 계정은 사용 권한만 부여받음. |
| **퇴사·담당자 변경** | 권한만 다른 계정으로 이관(RQ-MEMBER-203). 회사 마스터 계정 자체는 유지. | 회사 마스터 계정 비밀번호를 변경하여 신규 담당자에게 재배포(RQ-MEMBER-301). |
| **법인 대표번호(RQ-CALLER-003)** | 회사 단위에 묶이고 권한 보유자만 사용 가능. | 회사 마스터에 등록 → 모든 하위 계정 자동 상속(RQ-MEMBER-303). |
| **권한 모델 복잡도** | RBAC/ACL 적극 사용. 유연하나 감사가 까다로움. | 트리 위치 = 권한이라 단순. 세분 권한 추가 시 후속 개선 필요. |
| **2차 인증 적용** | 계정 단위 일률 적용. | 회사 마스터/하위 정책 분리 가능. |

|---|---|---|
| **결제 흐름** | 선불 충전 + 잔액 임계치 도달 시 자동 결제. | 선불 충전 단일. 임계치 알림(RQ-PAY-301~304)만 운영. |
| **PG 연동** | 정기/자동 결제 가능 PG 계약 필수. PG 심사 통과·30일 제한 옵션 적용. | 단건 결제만 지원. PG 심사 부담 낮음. |
| **보안 요구** | 한도·이상결제 알림·잠금 정책 필수. | 자동결제 없음 → 해킹 시 즉시 결제 피해 위험 낮음. |
| **운영 부담** | 결제 실패 재시도, 환불 응대 케이스 증가. 운영자 모니터링(RQ-ADMIN-509) 필요. | 잔액 부족 발송 중단 클레임 증가 가능. |
| **데이터 모델** | `auto_charge_setting` 테이블 신설. | `balance_alert_setting`만 운영. |

### §C. 구독형 결제 도입 여부 (RQ-PAY-401~409)

| 항목 | A안 — 도입 | B안 — 미도입 |
|---|---|---|
| **과금 방식** | 등급별 정액 + 포인트 차감(SMS 1pt / LMS 3pt / MMS 10pt). | 건당 단가 × 발송 건수 캐시 차감. |
| **데이터 모델** | `subscription_plan`, `subscription`, `point_balance`, `cash_balance` 분리. 차감 우선순위 정의. | `cash_balance` 단일. |
| **세금/회계** | 결제 주기 단위 세금계산서. 잔여분 환불·일할 계산 정책 신규. 사용금액 발급 원칙(RQ-PAY-505)과 충돌 가능. | 사용 금액 기준 발급 단순. |
| **운영 부담** | 환불·정책 문의 응대 부담 大. 운영자 등급 관리(RQ-ADMIN-511) 필요. | 단순. |
| **시장 포지셔닝** | 업계 선례 부재 — 선두 주자 가능. | 시장 표준. |

### §D. 카카오 직접 대행사 등록 (RQ-SEND-307, RQ-ADMIN-401 영향)

| 항목 | A안 — 직접 대행사 등록 | B안 — 중계사 연동 유지 |

| **운영 책임** | 카카오 정책 변경·템플릿 심사 직접 부담. | 중계사가 일부 흡수. |
