# MCP 문자 서비스 요구사항 명세서

> 작성일: 2026-04-29 (최종 수정 2026-04-30, v20)
> 상태: 1차 초안

## 사용자 역할 정의

본 명세서에서 사용하는 사용자 역할은 다음과 같이 명확히 분리한다.

| 역할 | 권한 코드 | 정의 | 권한 범위 |
|---|---|---|---|
| **비회원** | — | 가입 전 사용자 | 시스템 권한 없음. 공개 페이지(회원가입·FAQ·SDK 다운로드 등) 익명 접근 + **체험 모드(`RQ-TRIAL-*`)** 사용 가능 (더미 데이터, 실발송·결제 차단) |
| **회원 — 개인** | `MEMBER` | 본인 인증으로 가입한 개인 사용자 | 자신의 발신번호·키·발송·결제 |
| **회원 — 회사 마스터** | `COMPANY_MASTER` | 사업자 회원사의 오너 계정 (회사 단위 1인 또는 다수, A/B안 분기) | 회사 단위 발신번호·결제·하위 계정 관리. 시스템 차원 권한은 없음 |
| **회원 — 회사 하위 계정** | `COMPANY_MEMBER` | 회사 마스터가 생성한 직원 계정 | 회사 마스터가 부여한 권한 범위 내 발송·키 발급 등 |
| **시스템 운영자 — 슈퍼 어드민** | `SUPER_ADMIN` | 본 서비스를 운영하는 회사의 총괄 관리자 | 모든 운영 권한 보유. 다른 운영자 계정 발급·권한 설정 |
| **시스템 운영자 — 일반** | `ADMIN` | 슈퍼 어드민이 권한을 위임한 운영 담당자 | 위임된 권한만 (심사·CS·환불 등 분담) |

> ⚠ 회사 단위 오너는 **회사 마스터(`COMPANY_MASTER`)**, 서비스 총괄은 **슈퍼 어드민(`SUPER_ADMIN`)** 또는 **일반 운영자(`ADMIN`)** 로 표기한다. 권한 코드는 `AAA_AAA` UPPER_SNAKE_CASE로 통일한다.

> SDK/MCP/CLI는 별도의 사용자 유형이 아니라 **인증 컨텍스트**다. 회원(`MEMBER` / `COMPANY_MASTER` / `COMPANY_MEMBER`)이 자신의 계정에서 발급한 API Key를 통해 SDK/MCP/CLI에 접근하며, 키 보유자의 권한·발신번호·캐시 잔액·결제 수단을 그대로 위임받는다. 별도 "외부 개발자" 역할은 두지 않는다.

## ID 체계

| Prefix | 분류 | 설명 |
|--------|-------|------|
| `RQ-AUTH` | 회원가입/로그인 | 회원가입, 본인인증, 서류 제출, 로그인, 2차인증 |
| `RQ-TRIAL` | 비회원 체험 | 비회원 대상 체험 모드 진입·더미 데이터·실발송 차단·가입 전환 |
| `RQ-MEMBER` | 회사 마스터/하위 계정 | 고객사 내부의 회사 마스터, 하위 계정 관리 |
| `RQ-ADMIN` | 시스템 운영자 | 서비스 총괄 — 회원 심사/승인, 발송 통제, 환불 처리, 운영 설정 |
| `RQ-CALLER` | 발신번호 | 발신번호 등록·관리·검증 |
| `RQ-KEY` | API Key | 키 발급·관리·전환·권한 |
| `RQ-SDK` | SDK | 라이브러리(Python 등) 다운로드, 인증 |
| `RQ-MCP` | MCP | AI Agent가 호출하는 MCP 서버 (가이드 조회 + 발송 도구) |
| `RQ-CLI` | CLI | 명령행 도구 (회원 키로 발송·조회) |
| `RQ-SEND` | 발송 | 채널별 메시지 발송, 발송 이력 |
| `RQ-TPL` | 카카오/RCS 템플릿 | 브랜드·템플릿 등록·이관 |
| `RQ-PAY` | 결제/캐시 | 충전, 자동충전, 후불, 구독, 환불, 세금 |
| `RQ-TEST` | 테스트 환경 | SDK/MCP/CLI 한정 테스트 키·채널별 발송 한도 |
| `RQ-SEC` | 보안/스팸 | 스팸 필터, 이상 패턴, 한도 차단 |
| `RQ-OPS` | 고객 응대 | 챗봇, FAQ, 1:1 문의 |
| `RQ-ARCH` | 시스템/아키텍처 | 채널 통합, 발송 흐름, 인코딩 |

엑셀 변환 시 ID 끝 3자리는 일련번호로 사용한다.

## 1. 회원가입 / 로그인 (RQ-AUTH)

### 1.5. 비회원 체험 모드 (RQ-TRIAL)

> 비회원이 회원가입 없이 시스템을 직접 조작해볼 수 있도록 더미 데이터 기반의 체험 모드를 제공한다. 회원과 동일한 UI/메뉴를 노출하되 실제 발송·결제·외부 시스템 연동은 차단하며, 데이터는 운영 영역과 격리된다.

| RQ-ID | 분류 | 요구사항명 | 요구사항 내용 | 비고 |
|---|---|---|---|---|
| RQ-TRIAL-001 | 체험 | 체험 모드 진입 | 비회원은 메인 페이지의 "체험하기" 버튼을 클릭하여 회원가입 없이 체험 모드로 즉시 진입할 수 있다. | 본인 인증·이메일 입력 불요 |
| RQ-TRIAL-002 | 체험 | 체험 세션 발급 | 시스템은 체험 진입 시 익명 체험 세션을 발급하고 더미 회원사 컨텍스트(더미 회원 ID·회사·등급)를 부여한다. | 운영 회원 테이블과 분리 |
| RQ-TRIAL-003 | 체험 | 더미 데이터 사전 적재 | 시스템은 체험 세션에 더미 발신번호·더미 API Key·더미 발송 이력·더미 카카오 템플릿/RCS 브랜드·더미 캐시 잔액·더미 결제 이력을 사전 적재하여 노출한다. | 회원 화면 학습용 |
| RQ-TRIAL-004 | 체험 | 회원 동일 UI 노출 | 비회원은 체험 모드에서 회원과 동일한 UI/메뉴(발신번호·키·발송·이력·템플릿·결제·마이페이지)를 탐색·조작할 수 있다. | 회원 전환 시 학습 효과 보존 |
| RQ-TRIAL-005 | 체험 | 실발송 차단 | 시스템은 체험 세션의 발송 요청을 받으면 실제 통신사·중계사로 송출하지 않고 가상 발송 결과(성공/실패/대기 시뮬레이션)를 즉시 반환한다. | 외부 비용·법적 리스크 차단 |
| RQ-TRIAL-006 | 체험 | 결제·환불 차단 | 시스템은 체험 세션의 결제·충전·환불·세금계산서 발급 등 금전 거래 요청을 차단하고 시뮬레이션 결과만 노출한다. | |
| RQ-TRIAL-007 | 체험 | 외부 시스템 연동 차단 | 시스템은 체험 세션에서 KISA 발신번호 사전 등록, 카카오 템플릿 심사 신청, RCS 브랜드 등록 등 외부 시스템에 영구 영향을 주는 요청을 차단한다. | 시뮬레이션만 |
| RQ-TRIAL-008 | 체험 | 데이터 격리 | 시스템은 체험 세션의 모든 행위가 운영 회원·발송·정산 데이터에 영향을 주지 않도록 격리한다. | 운영 DB와 물리/논리적 분리 |

### 5.3. API Key 권한 스코프 (Permission Scopes)

> API Key 발급·수정 시 회원은 "이 키로 무엇을 할 수 있는가"를 **스코프 코드** 단위로 다중 선택한다. 스코프는 CLI 명령 그룹과 1:1로 매칭되어 직관적으로 관리되며, SDK·MCP·CLI 모든 진입점에서 동일하게 적용된다.

#### 5.3.1. 스코프 카탈로그

| 스코프 코드 | CLI 매핑 | 설명 |
|---|---|---|
| `send` | `wsc send sms / lms / mms / kakao / rcs` | 모든 채널 발송 (캐시 차감 발생) |
| `send:sms` | `wsc send sms` | SMS 단문만 발송 |
| `send:lms` | `wsc send lms` | LMS 장문만 발송 |
| `send:mms` | `wsc send mms` | MMS 이미지 포함 발송 |
| `send:kakao` | `wsc send kakao` | 카카오 알림톡만 발송 |
| `send:rcs` | `wsc send rcs` | RCS 메시지만 발송 |
| `history:read` | `wsc history list / get / 검색 옵션` | 발송 이력 목록·상세·필터 검색 (조회 범위는 RQ-KEY-203 적용) |
| `caller:read` | `wsc caller list` | 등록된 발신번호 목록·상태 조회 |
| `caller:manage` | `wsc caller register / delete` | 발신번호 등록 신청·삭제 (CLI·웹 명시 명령 전용 — MCP 비노출 RQ-MCP-014) |
| `key:read` | `wsc key list` | 자신의 API Key 목록·환경 조회 |
| `balance:read` | `wsc balance / wsc usage` | 캐시 잔액·사용 내역·테스트 발송 한도 잔여 조회 |
| `template:read` | `wsc template list` | 카카오 템플릿 목록·심사 상태 조회 |
| `brand:read` | `wsc brand list` | RCS 브랜드 목록 조회 |

#### 5.3.2. 조회 범위 스코프 (Read Scope)

`history:read` 등 조회 계열 스코프에 대해 회원은 키별로 조회 범위를 지정한다.

| 코드 | 동작 |
|---|---|
| `scope:key` (기본값) | API Key 인증으로 호출 시, **그 키로 직접 발송한 건만** 조회 가능 |
| `scope:member` (옵션) | 회원이 명시 활성화 시, 같은 회원의 **다른 키 / 웹 콘솔 발송 포함 전체** 조회 가능 |

기본값 `scope:key`로 둠으로써 키 유출 시 노출 범위를 축소한다. 회사 마스터의 통합 모니터링 키 등에 한해 `scope:member`를 명시 활성화한다.

#### 5.3.3. 스코프 운영 요구사항

| RQ-ID | 분류 | 요구사항명 | 요구사항 내용 | 비고 |
|---|---|---|---|---|
| RQ-KEY-201 | 권한 | 스코프 카탈로그 노출 | 시스템은 API Key 발급·수정 화면에서 §5.3.1 카탈로그를 다중 선택 UI로 노출하고, 각 코드의 설명·CLI 매핑·캐시 차감 여부를 함께 표시한다. | 회원이 의미를 이해한 상태로 선택 |
| RQ-KEY-202 | 권한 | 스코프 미허용 호출 차단 | 시스템은 API Key에 부여되지 않은 스코프의 호출(SDK·MCP·CLI 모두)을 차단하고 거부 사유(필요 스코프 코드)를 응답한다. | |
| RQ-KEY-203 | 권한 | 키별 조회 범위 토글 | 회원은 키별로 조회 범위 스코프를 `scope:key`(기본) 또는 `scope:member`로 설정할 수 있다. | 조회 계열 스코프(`history:read` 등)에 적용 |
| RQ-KEY-204 | 권한 | 스코프 변경 이력 조회 | 회원은 자신의 키별 스코프·조회 범위 부여·회수·변경 이력을 조회할 수 있다. | 키 보안 추적 — 유출 의심 시 침해 범위 파악 |
| RQ-KEY-205 | 권한 | 권장 스코프 프리셋 | 시스템은 키 발급 시 용도별 권장 프리셋(예: "발송 전용", "조회 전용", "통합 모니터링")을 제시하여 회원이 빠르게 시작할 수 있게 한다. | 회원 경험 개선 — 카탈로그를 일일이 고르지 않아도 됨 |

## 6. 외부 통합 도구 (SDK / MCP / CLI)

> 외부 시스템이 본 서비스를 자기 시스템에 통합할 때 사용할 수 있는 3가지 진입점이다. 모든 진입점은 **회원이 발급한 API Key로 인증**되며, 키 보유자(회원)의 권한·한도·결제 수단을 그대로 위임받는다.

### 왜 SDK·MCP·CLI를 모두 제공하는가 — 인프라 통합 / CLI 우선 전략

같은 작업이라도 어떤 진입점으로 호출하느냐에 따라 **AI 토큰 소모가 크게 달라지고, 통합 환경(코드·AI Agent·셸)에 맞는 최적 도구가 다르다.** 본 서비스는 세 진입점을 함께 제공하여 회원이 워크플로우 특성에 맞는 도구를 골라 쓸 수 있게 한다.

| 진입점 | 호출 주체 | 응답 형식 | 메타데이터 | 토큰 소모(AI Agent 호출 시) | 설치/설정 | 커스터마이징 | 권장 시나리오 |
|---|---|---|---|---|---|---|---|
| **SDK** | 자사 백엔드/프론트엔드 코드 | 언어 네이티브 객체 | 언어 타입 시스템에 의존 | 해당 없음 | 패키지 매니저로 한 줄 | 자기 코드에서 자유 가공 | 자기 서비스에 import해서 호출 |
| **MCP** | 사용자의 AI Agent | 구조화된 JSON (스키마 강제) | **풍부함** (전체 메타·스키마·페이로드) | **높음** | 간단 (AI Agent 클라이언트 설정만) | 제한적 | 일회성·탐색적·자연어 질의 ("어제 발송 결과 봐줘") |
| **CLI** | 셸·스크립트·CI/CD·AI Agent의 셸 호출 | 텍스트(기본) 또는 JSON(`--output json`) | **선택적 — 필요한 출력만** | **낮음** | 중간 (바이너리 + 인증 1회) | 자유 (필터·포맷·파이프) | 반복 작업·정형 처리·비용 민감 워크플로우 |

#### 토큰 경제학 (실측 비교)

> **Jira 티켓 1건 조회 기준 — MCP 호출: 약 4,000~6,000 토큰 / CLI(`jira issue view --plain`) 호출: 약 600~900 토큰. 5~7배 차이.**
>
> 하루 50회 호출 기준으로도 누적 차이가 크고, 자동화·CI/CD·반복 모니터링 시나리오일수록 격차가 벌어진다(10배 가까이 벌어지는 케이스도 보고됨). 동일한 비교가 본 서비스의 발송 이력·잔액 조회 등에도 그대로 적용된다.

> ⚠️ 이 토큰 비용은 **사용자 본인의 AI Agent에서 발생**하는 것이며 본 서비스가 측정·과금하지 않는다(v5에서 본 서비스 토큰 과금 컨셉 폐기). 다만 사용자의 AI Agent 비용 절감을 위해 CLI를 함께 제공하는 것이 핵심 설계 의도다.

#### 권장 사용 분리

- AI Agent로부터의 **자연어 단발 명령** ("이번 달 발송 통계 알려줘") → **MCP**
- AI Agent가 **자동화 루프 안에서 같은 작업을 반복**하거나 대량 처리 → AI Agent 안에서 **CLI 호출로 위임** (예: `!wsc history list --output json`) — 토큰 5~10배 절감
- **자기 백엔드 코드에서 직접 호출** → **SDK**
- **셸 스크립트·서버 cron·CI 파이프라인** → **CLI**

### 인증·과금 원칙

- **다운로드/설치/정적 가이드 조회는 누구나 가능** (SDK pip/npm, CLI 바이너리, `wsc docs` 등).
- **발송·조회·잔액 등 자원 호출은 API Key 필수** (회원 전용).
- **과금은 메시지 발송 시 캐시 차감만 발생**. SDK 다운로드·MCP 가이드 조회·CLI 설치/docs/snippet은 무과금.
- **MCP 호출에서 사용자가 쓰는 AI 토큰은 사용자 자신의 AI Agent 비용**이므로 본 서비스 책임 범위 외이며, 측정·표시·과금하지 않는다.

### 진입점별 인증 동작

- **SDK** — 자기 코드 안에 import 해서 호출하는 라이브러리. 패키지는 공개 배포되어 누구나 다운로드 가능, 발송 호출 시 API Key 인증 필요.
- **MCP** — 사용자의 AI Agent가 본 서비스 MCP 서버에 연결하여 가이드 조회·발송 도구를 호출. 서버 자원 보호를 위해 가이드 조회를 포함한 모든 호출에 API Key 인증 필요.
- **CLI** — 셸 명령으로 호출하는 도구. 바이너리는 공개 배포, `--help`/`docs`/`snippet` 등 정적 가이드는 무인증, 발송·조회는 API Key 인증 필요.

### 7.4. 발송 시스템 연계

> **본 서비스의 발송 책임 범위.** 본 서비스는 별도로 구축된 외부 **발송 시스템**(발송 에이전트 + 통신사·중계사 연동 인프라)을 직접 구현하지 않는다. 본 서비스의 책임은 회원·외부 채널(SDK/MCP/CLI)의 발송 요청을 받아 **정합성 검증 → 인코딩 변환 → 발송 테이블 적재**까지이며, 그 이후의 채널 라우팅, 중계사 1:1 매핑 적용, 폴백 실행, 실제 송출은 외부 발송 시스템이 발송 테이블을 읽어 수행한다.

> **회원 노출 추상화 원칙.** 회원은 카카오·RCS 등 채널의 **중계사·라우팅·내부 채널 식별자를 알 수 없고 알 필요도 없다**. 회원이 인지하는 채널은 "SMS / LMS / MMS / 카카오 알림톡 / RCS"까지로 한정되며, "어느 중계사를 거쳐 발송되는가"는 본 서비스 운영자(RQ-ADMIN-401)만 다루는 내부 정보다.

### §A. 회사 마스터 권한 구조 (RQ-MEMBER-201~303)

> 본 분기는 **회사(고객사) 내부의 마스터 계정 모델**에 대한 의사결정이며, 시스템 운영자(`RQ-ADMIN-*`)와는 무관하다.

| 항목 | A안 — 권한 부여형 | B안 — 트리 종속형 |
|---|---|---|
| **개념** | 회사 내에 별도의 마스터 계정을 두지 않고, 일반 계정 중 특정 사용자에게 회사 마스터 권한을 부여한다. | 회사 마스터 계정 아래 하위 계정이 트리 구조로 종속된다. |
| **DB 스키마** | `accounts` 단일 테이블 + `account_role`(주체, 역할) 다대다 매핑. 회사 식별은 `company_id` 컬럼. | `companies` 1차 테이블 + `accounts.company_id` FK + `accounts.is_company_master` 플래그. 트리 깊이 1단 고정. |
| **발신번호·키 종속** | 발신번호·API Key는 `company_id` 단위 + 등록자(`account_id`) 기록. 권한 보유자만 사용. | 발신번호·API Key는 회사 마스터 계정에 직접 종속. 하위 계정은 사용 권한만 부여받음. |
| **퇴사·담당자 변경** | 권한만 다른 계정으로 이관(RQ-MEMBER-203). 회사 마스터 계정 자체는 유지. | 회사 마스터 계정 비밀번호를 변경하여 신규 담당자에게 재배포(RQ-MEMBER-301). |
| **법인 대표번호(RQ-CALLER-003)** | 회사 단위에 묶이고 권한 보유자만 사용 가능. | 회사 마스터에 등록 → 모든 하위 계정 자동 상속(RQ-MEMBER-303). |
