검증 내역
  • SendValidationGate 인터페이스 + order() 메서드로 실행 순서 제어 설계 확인
    order() 오름차순 정렬 후 첫 실패에서 즉시 예외 — fail-fast 파이프라인
  • SendValidationContext record — totalCost() 계산, 불변 입력 객체 구조 확인
    recipientCount × unitCost = totalCost, BalanceGate에서 사전 평가에 사용
  • SendErrorCode 카탈로그 13종 — 발신번호·스코프·한도·스팸·광고·잔액·API Key 상태 분류
    02_FEATURE_SPEC §6.1 예외 목록 및 §11 잔액 분기 기반
  • BalanceGate Redis 캐시 전략 확인 — write-around 패턴, TTL 30초, 캐시 미스 시 DB 직접 조회
    balance:{memberId} 키, CHARGE_BALANCE 테이블 SUM(amount_remaining) 집계
  • NightAdBlockGate isNightTime() — 경계값 21:00, 07:59, 08:00, 20:59 모두 단위 검증
    정보통신망법 §50조의8 준수, KST(Asia/Seoul) 기준 적용
  • PostpaidBlockGate 3단계 조회 — company_id → billing_mode → OVERDUE 청구서 COUNT
    개인 회원(company_id null) / 선불 회사 / 후불 연체 분기 모두 검증
  • SendValidationService — 9종 게이트 Spring 자동 주입, 역순 등록 후 order() 정렬 동작 확인
    오케스트레이터 통합 테스트 6케이스 — 전체 통과 / 첫 실패 / 중간 실패 / 마지막 실패
  • 로컬 빌드 실행 검증 불가 — SSL 인터셉션 환경
    PKIX 오류로 gradle 외부 다운로드 실패. 컴파일 정합 확인 후 CI 위임

결론 및 잔여 이슈

W-201 발송 정합성 검증 9종이 모두 구현되고 단위 테스트로 충족이 확인되었습니다. SendValidationGate 인터페이스 기반 파이프라인은 order() 오름차순 정렬과 fail-fast 실행을 보장하며, Spring 자동 주입으로 게이트 추가·제거가 오케스트레이터 코드 변경 없이 가능합니다. BalanceGate의 Redis 캐시 write-around 패턴(TTL 30s)과 PostpaidBlockGate의 3단계 조회 구조가 핫 패스 성능을 유지하면서 정합성을 보장합니다. W-202(인코딩 변환), W-203(발송 테이블 적재)의 블록이 해소되어 후속 태스크 착수가 가능합니다.

잔여 이슈 (다음 스프린트 또는 별도 태스크)
  • SpamKeywordGate 키워드 목록 — MVP 하드코딩, 운영 환경에서 DB 테이블 또는 외부 스팸 필터 API로 교체 필요
  • DailyLimitGate Redis 카운터 증감 — 현재 검증만 수행, 발송 완료 후 카운터 INCR은 W-203(적재) 단계에서 구현
  • BalanceGate 잔액 차감 — 현재 사전 평가만 수행, 실제 차감은 발송 완료 이벤트 처리 단계에서 수행
  • 로컬 빌드 실행 검증 — SSL 인터셉션 환경으로 gradle 외부 다운로드 불가, CI 파이프라인 통합 후 확인
  • NightAdBlockGate 예약 발송 — 예약 시각 기준 야간 차단 로직은 W-206(예약 발송) 단계에서 별도 처리 필요
