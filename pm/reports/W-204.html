합의 스펙 요약 (05_DATA_MODEL §5 기준)

1. 발송 테이블 컬럼 스펙 (send_*_tran INSERT 시 본 서비스 책임 컬럼)

컬럼 타입 NULL 합의 내용
msg_type VARCHAR(4) NOT NULL 채널 매핑: SMS="SMS" / LMS="LMS" / MMS="MMS" / KAKAO="AT" / RCS="RCS"
msg_sub_type VARCHAR(5) NOT NULL SMS/LMS/MMS: 광고="AD" / 일반="NORM". KAKAO="AT". RCS: 템플릿="TMPL" / 일반텍스트="TEXT"
callback VARCHAR(20) NOT NULL 정규화된 발신번호 (E.164 또는 국내 형식). W-201 등록 검증 통과 값
message_state TINYINT NOT NULL 본 서비스 INSERT 시점에 항상 0(init). 이후 전이는 외부 시스템 책임
request_date DATETIME NOT NULL 즉시 발송: 현재 시각. 예약 발송: 회원 지정 미래 시각. 외부 시스템이 도래 시점에 자동 송출
group_id INT NULL 일괄 발송 묶음 식별자. 단건은 NULL. W-203에서 UUID 기반 생성
user_id BIGINT NOT NULL 내부 memberId → 외부 user_id 1:1 매핑
etc_char_1 VARCHAR(26) NULL 내부 send_id(ULID 26자) 보관. msg_id ↔ sendId 역추적용. 회원 비노출
etc_char_2 VARCHAR(500) NULL INV-02 라우팅 메타(중계사 매핑). 회원·API 응답·CLI·MCP 어디에도 절대 비노출
fb_msg TEXT NULL 카카오·RCS 1차 실패 시 외부 시스템 자동 폴백용 SMS/LMS 본문. 그 외 채널 NULL

2. message_state 카탈로그 (§5.3)

값 코드 보유 테이블 설명 및 책임
0 init _tran 본 서비스 INSERT 직후 초기값. 외부 시스템이 아직 처리하지 않은 상태
1 fetched _tran 외부 시스템이 송출 큐로 가져감. 전이 책임: 외부 시스템
2 submitted _tran 외부 시스템 통신사 제출 성공. 전이 책임: 외부 시스템
3 finished _tran → _log_YYYYMM 처리 완료. 로그 테이블 이관 대상. polling 종료 조건
4 logfail _tran 로그 이관 실패. 외부 운영자 점검 대상. polling 종료 조건

3. polling 규약

조회 대상: send_*_tran 우선 → 없으면 당월 send_*_log_YYYYMM → 이전 월 순 탐색
조회 키: msg_id (외부 시스템 발급, ExternalSendResult.msgId 로 수신)
주기 — 테스트망: 10초 간격
주기 — 상용망: 최초 30초 후 → 이후 60초 간격
종료 조건: message_state 3(finished) 또는 4(logfail) 도달
성공 판정: message_state=3 + result_code="0000"
결과 캐시 (옵션): Redis send:result:{msg_id} TTL 5분 (§5.6 검토 사항)
외부 발송 연동 흐름 (W-204 범위)
1
W-203 SendRequest 적재 완료 (status=PENDING)
sendId(ULID) 발급 완료. externalMsgId=NULL, routingMeta=NULL 상태
2
ExternalSendRecord 생성
ExternalSendRecord.from(req, routingMeta, fbMsg) — SendRequest 엔티티를 외부 테이블 INSERT 페이로드로 변환. msgType/msgSubType 채널·광고여부 기반 산출. etcChar1=sendId, etcChar2=routingMeta(회원 비노출)
3
ExternalDispatchAdapter.insert(record) 호출
스텁: 순번 msg_id 즉시 반환. 실 환경: 외부 DB JDBC INSERT 또는 HTTP API 호출로 교체
4
INSERT 결과에 따른 상태 전이
성공: req.markQueued(msgId) → status=QUEUED, externalMsgId 업데이트
실패: req.markFailed(reason) → status=FAILED (보상 트랜잭션은 W-205 이후 처리)
5
ExternalDispatchAdapter.poll(msgId) — 결과 polling
messageState 3/4 도달까지 주기적 조회. ExternalDispatchPollingResult.isTerminal()로 종료 판정. 호출 주기는 호출자가 관리(테스트망 10s / 상용망 60s)

결론 및 잔여 이슈

W-204 외부 발송 시스템 인터페이스 합의·테스트를 완료했다. 발송 테이블 컬럼 스펙(send_*_tran INSERT 책임 컬럼 전체), polling 방식·주기·종료 조건, message_state 카탈로그(0~4), 상태 회신 규약을 코드 주석·JavaDoc 형태로 문서화했다. Adapter 패턴으로 외부 시스템 연동 경계를 ExternalDispatchAdapter 인터페이스로 격리했으며, 테스트·개발 환경용 스텁(StubExternalDispatchAdapter)과 서비스(ExternalDispatchService)를 구현해 W-203 SendRequest.markQueued() 연계 흐름을 단위 테스트로 검증했다. 로컬 SSL 인터셉션 환경 제약으로 빌드 실행 대신 컴파일 정합·Mockito 단위 테스트로 동작을 검증했다.

잔여 이슈 (후속 스프린트에서 처리 예정)
  • 실제 중계사 JDBC/HTTP 구현체 미구현 — StubExternalDispatchAdapter 를 실 구현체로 교체 시 인터페이스 계약만 유지하면 서비스 코드 변경 없음
  • 잔액 차감(§7 CHARGE_BALANCE_LEDGER) 미연계 — 외부 INSERT 직전 차감·실패 시 REVERT 보상 트랜잭션은 W-205 이후 처리
  • routingMeta 산출 로직 미구현 — Redis routing:{memberId} 캐시 기반 중계사 매핑 값 조회 후 ExternalSendRecord.etcChar2 에 주입하는 로직 필요
  • 폴백(fb_msg) 페이로드 생성 미구현 — 카카오·RCS 채널의 대체발송 SMS/LMS 본문 산출 로직(§5.5 대체발송)
  • polling 스케줄러 미구현 — 현재는 poll() 단발 호출만 가능. 주기적 polling 스케줄러(@Scheduled 또는 비동기 큐) 구현 필요
  • result_code 카탈로그 — 통신사별 에러 코드 전체 목록은 외부 시스템 합의(R-01) 후 §5에 추가 표기 예정
