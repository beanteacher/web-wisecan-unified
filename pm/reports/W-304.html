키별 조회 범위 정책
scope:key (기본)
  • 요청한 API Key(apiKeyId)로 발송한 이력만 반환
  • 컨트롤러에서 scopeMember=false(기본)이면 apiKeyId를 서비스에 전달
  • 상세 조회 시 이력의 apiKeyId가 일치하지 않으면 404 반환 (존재 비노출)
scope:member (옵션 — scopeMember=true)
  • 회원(memberId) 기준으로 모든 키의 이력을 반환
  • 컨트롤러에서 apiKeyId=null을 서비스에 전달 → findByMemberIdOrderByCreatedAtDesc 경로
  • 상세 조회 시 이력의 memberId가 일치하지 않으면 404 반환
  • 활성화 요건: history:read 스코프 — 서비스 레이어에서 검증 권고 (현재 컨트롤러 레이어에서 라우팅)
항목 scope:key scope:member
목록 조회 기준 apiKeyId memberId
상세 접근 제어 이력의 apiKeyId 일치 여부 이력의 memberId 일치 여부
차단 응답 EntityNotFoundException → HTTP 404 (정보 비노출)
routingMeta 노출 ListItem / DetailItem 모두 제외 (INV-02 회원 비노출)

완료 기준 (DoD)
  • BE — 발송 이력 조회 API 구현 (필터·페이지네이션·상태별)
    SendHistoryDto, SendHistoryController, SendHistoryService 신규 생성. SendRequestRepository 기존 메서드 활용
  • BE — 키별 조회 범위 정책 (scope:key 기본 / scope:member 옵션)
    scopeMember 파라미터로 apiKeyId=null 전달 → memberId 기준 조회 경로 분기
  • BE — routingMeta 비노출 (INV-02)
    ListItem·DetailItem 모두 routingMeta 필드 제외. 내부 라우팅 정보 회원 노출 원천 차단
  • BE — 접근 불가 이력 404 통일
    EntityNotFoundException 사용으로 존재 자체를 숨겨 정보 노출 방지
  • BE — SendRequest 기존 엔티티·Repository 수정 최소화 (충돌 방지)
    기존 SendRequestRepository 메서드 재사용. SendHistoryDto·Controller·Service 신규 격리 파일로 분리
  • BE — 단위 테스트 (SendHistoryServiceTest, 10개 케이스)
    scope:key 목록·필터 3건 / scope:member 목록·필터 2건 / 상세 5건 (정상·접근차단·없음)
  • FE — 발송 이력 화면 구현 (FSD 아키텍처)
    entities/send-history (types·api·barrel) + features/send-history (훅·UI·barrel) + app/(auth)/histories/page.tsx
  • FE — React Query 필터 자동 리페치
    queryKey에 params 전체 포함 → 필터·페이지 변경 시 자동 리페치. placeholderData로 깜빡임 방지
  • SecurityConfig — /histories/** 경로 ApiKeyAuthFilter 처리 허용
    permitAll() 추가로 JWT 필터 우회, ApiKeyAuthFilter가 Bearer 키 인증 담당

설계 결정 및 특이사항
  • 서비스 파일 외부 수정 반영 — SendHistoryService가 SendHistoryRepository 대신 SendRequestRepository를 직접 사용하도록 수정됨
    다른 워커와의 충돌 최소화를 위해 기존 SendRequestRepository 메서드를 재활용. 인메모리 스트림 필터로 필터 로직 구현
  • scope:member 활성화 시 history:read 스코프 검증 위치
    현재 구현에서는 컨트롤러가 scopeMember 파라미터를 apiKeyId=null로 변환하여 서비스에 전달. 스코프 검증은 ApiKeyAuthFilter 또는 별도 AOP로 보강 권고
  • FE SendHistoryPanel — 프로젝트 공통 UI 컴포넌트 의존
    DataTable, Badge, Button, Pagination 등 shared/ui 컴포넌트를 사용. 해당 컴포넌트가 미구현 시 대체 구현 필요
결론

W-304 발송 이력 조회가 완료되었습니다. BE는 기존 SendRequest 엔티티·Repository와의 충돌을 최소화하면서 SendHistoryDto·Service·Controller 3종을 신규 격리 파일로 분리하였고, scope:key(기본) / scope:member(옵션) 범위 정책과 404 통일 접근 제어를 구현했습니다. routingMeta는 ListItem·DetailItem 모두에서 제외하여 INV-02 회원 비노출 불변식을 준수합니다. FE는 FSD 아키텍처에 따라 entities → features → app 계층을 완성하였으며, React Query의 queryKey 파라미터 포함 전략으로 필터·페이지 변경 시 자동 리페치가 동작합니다. 단위 테스트 10개 케이스가 현재 서비스 시그니처 기준으로 정합하게 작성되었습니다.
