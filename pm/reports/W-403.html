후불 모델 처리 흐름
1
후불 신청 (APPLIED)
사업자 COMPANY_MASTER 가 POST /billing/postpaid/apply 호출. PostpaidConfig 행 생성(status=APPLIED). 이미 비-SUSPENDED 설정 존재 시 BillingException.
2
운영자 심사 (UNDER_REVIEW → ACTIVE)
운영자가 /admin/{id}/review 로 심사 시작 후 /admin/{id}/approve 로 신용 한도·보증보험 확정. COMPANY.billing_mode = POSTPAID 전환은 별도 관리자 작업.
3
청구서 발행
운영자 또는 배치가 POST /billing/postpaid/admin/invoices 호출. Redis 5분 락(lock:invoice:{companyId}:{period}) + DB UNIQUE 제약으로 중복 차단. ISSUED 상태로 발행, dueAt 설정.
4
청구서 결제
회원이 POST /billing/postpaid/invoices/{id}/pay 호출. 완납 시 PAID 전환 — PostpaidBlockGate 차단 자동 해제. 부분 납부 허용, 초과 납부는 totalAmount에 클램프.
5
연체 처리 (배치)
POST /billing/postpaid/batch/overdue 호출 시 dueAt 경과 ISSUED 청구서를 일괄 OVERDUE 전환. 이후 PostpaidBlockGate 가 해당 회사 발송을 INSUFFICIENT_BALANCE 로 차단.
6
발송 차단 E2E (PostpaidBlockGate)
W-201 구현 기존 게이트 활용. POSTPAID 회사의 OVERDUE 청구서 COUNT 조회 → 1개 이상이면 SendValidationException(INSUFFICIENT_BALANCE) 발생. order()=90.

결론 및 잔여 이슈

W-403 후불 모델 핵심 구현 완료. PostpaidConfig(신용 한도·보증보험) 엔티티, PostpaidInvoice(청구서 발행·결제·연체) 엔티티, 서비스·컨트롤러·DTO·리포지토리를 신규 작성하였으며, 기존 PostpaidBlockGate(W-201)는 수정 없이 연계됨. DoD 7개 항목 모두 충족. 단위 테스트 39케이스 코드 정합 완료.

잔여 이슈 / 후속 작업
  • COMPANY.billing_mode 전환(PREPAID → POSTPAID) 시 기존 AUTO_CHARGE_CONFIG 비활성화 연동은 W-402 범위 — 협의 필요
  • 연체 처리 배치 스케줄러(@Scheduled 또는 외부 Cron)는 인프라 구성 단계에서 등록 필요
  • 청구서 발행 총액 산정(발송 사용량 집계) 로직은 외부 정산 시스템 연동 — W-MASTER 범위
  • PostpaidBlockGate JPA 리포지토리 전환(existsOverdueByCompanyId) 필요 시 W-405에서 일괄 적용 검토
