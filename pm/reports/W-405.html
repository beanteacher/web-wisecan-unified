4분기 분기 로직 (02_FEATURE_SPEC §11.1)
1
자동결제 (AutoCharged)
HTTP 200
AutoChargeConfig.isEnabled() = true + expiresAt 미만 → AutoChargeService.triggerIfNeeded() 호출. DB에서 잔액 재조회하여 부족액이 해소되면 AutoCharged(chargedAmount) 반환. 개인 회원·법인 모두 적용.
2
후불 (Postpaid)
HTTP 200
자동결제 비활성 + companyId != null + PostpaidConfig.isActive() = true → Postpaid(deferredAmount) 반환. companyId = null(개인 회원)은 이 분기를 건너뜀. PostpaidStatus.ACTIVE 상태 직접 확인 (isActive() 메서드).
3
부분 발송 (Partial)
HTTP 207
자동결제·후불 모두 비활성 + partialChoice = true → currentBalance / unitCost 건 수락, 나머지 rejectedNumbers 포함 Partial 반환. PartialResponse: acceptedCount, rejectedCount, rejectedNumbers, rejectReason("INSUFFICIENT_BALANCE"), shortfall.
4
전체 취소 (Cancelled)
HTTP 402
자동결제·후불 모두 비활성 + partialChoice = false → InsufficientFundsException(shortfall) throw → GlobalExceptionHandler HTTP 402 Payment Required. 응답 바디: ApiResponse.error("잔액이 부족합니다. 부족액: {N}원").
InsufficientBalanceRouter.route() 흐름
0
DB 직접 조회 (캐시 우회)

BalanceQueryService.getBalanceFromDb(memberId) — Redis 캐시 부정합 방지

1
잔액 충분 여부 선체크 (race-condition 방어)

currentBalance >= totalCost → AutoCharged(0) 즉시 반환 — 분기 진입 불필요

2
자동결제 설정 조회

AutoChargeConfigRepository.findByMemberId() + isEnabled() → 활성이면 triggerIfNeeded() 후 AutoCharged

3
후불 설정 조회 (companyId != null 조건 선행)

PostpaidConfigRepository.findByCompanyId() + isActive() → 활성이면 Postpaid(totalCost)

4
partialChoice 분기

true → 수락 가능 건 수 계산 후 Partial / false → Cancelled(shortfall)

X
Cancelled → InsufficientFundsException

컨트롤러 buildResponse()에서 throw — GlobalExceptionHandler가 HTTP 402로 변환

핵심 설계 결정
  • sealed interface BalanceBranchResult (Java 21)
    4개 permitted record — AutoCharged / Postpaid / Partial / Cancelled. 컴파일 타임에 모든 분기 처리를 강제하는 exhaustive switch 패턴 적용.
  • BalanceGate 비침습적 설계
    기존 SendValidationService(order=80) 의 INSUFFICIENT_BALANCE 예외 구조를 건드리지 않음. InsufficientBalanceRouter는 발송 전 별도 엔드포인트로 분리하여 호출 순서 변경 없이 기능 추가.
  • DB 직접 조회로 정합성 보장
    BalanceQueryService.getBalanceFromDb() 사용 — 잔액 부족 판단은 Redis 캐시를 우회하여 TTL 30s 스테일 리드 위험 제거.
  • companyId = null 개인 회원 안전 처리
    후불 분기(PostpaidConfig 조회) 진입 전 companyId null 체크를 선행 — NPE 없이 분기 2를 건너뜀.
  • HTTP 207 Multi-Status — 부분 성공 표현
    RFC 4918 준수. acceptedCount / rejectedCount / rejectedNumbers / shortfall 필드로 클라이언트가 거부 수신자를 재처리할 수 있도록 설계.

결론 및 잔여 이슈

W-405는 02_FEATURE_SPEC §11.1의 4분기 잔액 부족 분기 로직을 Java 21 sealed interface 패턴으로 타입 안전하게 구현했습니다. BalanceBranchResult(AutoCharged / Postpaid / Partial / Cancelled) 4종을 exhaustive switch로 처리하여 분기 누락을 컴파일 타임에 방지합니다. 기존 SendValidationService(BalanceGate)를 건드리지 않고 별도 엔드포인트를 추가하는 비침습적 설계를 채택했으며, DB 직접 조회로 Redis 캐시 부정합 위험을 제거했습니다. 단위 테스트 5개(4분기 + race-condition 경계)가 모든 분기 시나리오를 독립적으로 검증하며, HTTP 207(부분 발송)과 HTTP 402(전체 취소) 응답 표준이 클라이언트 재처리 경로를 명확히 제공합니다.
